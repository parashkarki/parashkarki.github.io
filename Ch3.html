Overview of SQL

Why SQL matters and where it came from

SQL (Structured Query Language) is the standard language used to create, read, update, and delete data in relational databases. Think of a database as a massive digital filing cabinet, and SQL is the language you use to ask it for exactly what you need.

🌎 Real Life Analogy

Imagine a giant library with millions of books. SQL is like the librarian's request form — you write exactly what book you want, from which shelf, sorted in which order, and the librarian (database) finds it for you instantly. Companies like Netflix, Amazon, and Google all use SQL databases behind the scenes every second.

🏦

Banking

Track every transaction and balance in real time

🛒

E-Commerce

Amazon uses SQL to store 350M+ product listings

🏥

Healthcare

Patient records, prescriptions, appointment scheduling

History of SQL

1970s
IBM developed "SEQUEL" (Structured English Query Language) as part of the System R research project at IBM San Jose Research Lab.
1986
SQL-86: ANSI and ISO officially standardize SQL, making it a recognized international standard.
1992
SQL-92: Major upgrade — the version most databases still base themselves on today.
1999
SQL:1999 — Added object-oriented features. Name changed from "SQL-99" to be Y2K compliant!
Today
MySQL, PostgreSQL, Oracle, SQL Server, MariaDB — all speak SQL. It's the most used query language in the world.

DDL — Building the Blueprint

Define what your data looks like before storing it

DDL is like designing the blueprint of a building before construction begins. You decide how many rooms (tables), what goes in each room (columns), and the rules for each room (constraints).

🌎 Real Life Analogy

Think of DDL like designing a Google Form. Before anyone fills it out, you decide: What questions exist? What type of answer is allowed (text, number, date)? Which questions are required? That's exactly what DDL does for a database table.

DDL lets you specify:

  • The schema — what columns each table has
  • The data type of each column (text, number, date…)
  • Integrity constraints — rules the data must follow
  • Indexes for fast search
  • Security — who can see and edit what

CREATE TABLE & Constraints

Building tables and enforcing data rules

🌎 Real Life Analogy

Creating a table is like designing a spreadsheet template for your company. You name the columns (ID, Name, Salary), set what type of data goes in each, and add rules: "ID must be unique," "Name can't be blank," "Department must exist in our department list."

💡 Example: Create an Instructor Table
CREATE TABLE instructor (
    ID          char(5),
    name        varchar(20) NOT NULL,
    dept_name   varchar(20),
    salary      numeric(8,2),
    PRIMARY KEY (ID),
    FOREIGN KEY (dept_name) REFERENCES department
);

The Three Key Constraints

🔑

PRIMARY KEY

Uniquely identifies each row — like your national ID number. No duplicates, never null.

🔗

FOREIGN KEY

Links to another table's primary key — like a passport referencing your country's records.

🚫

NOT NULL

The field must always have a value — like a form field marked as "Required".

Updating Table Structure

-- Add a new column
ALTER TABLE instructor ADD email varchar(50);

-- Remove a column
ALTER TABLE instructor DROP email;

-- Delete ALL data from a table (but keep the table)
DELETE FROM student;

-- Delete the table entirely
DROP TABLE instructor;

The SELECT Clause

Choosing which columns to display in results

🌎 Real Life Analogy

SELECT is like choosing which columns to show in a spreadsheet. Imagine an Excel sheet with 20 columns about employees. You only want to see "Name" and "Salary" — you tell Excel to show just those two. That's SELECT.

Basic Structure

SELECT A1, A2, ..., An
FROM   r1, r2, ..., rm
WHERE  condition;

Get all instructors' names

SELECT name
FROM   instructor;
💡 SQL is Case-Insensitive

select = SELECT = Select. Convention is to write keywords in UPPERCASE for readability.

Removing Duplicates with DISTINCT

🌎 Real Life Analogy

Imagine asking "Which cities do our employees live in?" You don't want "New York" listed 500 times — just once. DISTINCT removes duplicates, like a unique list filter.

-- Without DISTINCT: shows "Comp. Sci." once per instructor
SELECT dept_name FROM instructor;

-- With DISTINCT: shows each department name only once
SELECT DISTINCT dept_name FROM instructor;

-- ALL keyword (default) keeps duplicates
SELECT ALL dept_name FROM instructor;

-- * means "all columns"
SELECT * FROM instructor;

-- Arithmetic: Show monthly salary
SELECT ID, name, salary/12 AS monthly_salary
FROM   instructor;
🎯

The WHERE Clause

Filtering rows to find exactly what you need

🌎 Real Life Analogy

WHERE is like the search filter on a shopping site. On Amazon, you filter: "Show me only headphones, under $50, with 4+ stars." Each filter condition narrows the results — that's exactly what WHERE does with your data.

-- Find instructors in Computer Science
SELECT name
FROM   instructor
WHERE  dept_name = 'Comp. Sci.';

-- Multiple conditions: CS instructors earning > $70,000
SELECT name
FROM   instructor
WHERE  dept_name = 'Comp. Sci.' AND salary > 70000;

-- BETWEEN: salary in a range (inclusive)
SELECT name
FROM   instructor
WHERE  salary BETWEEN 90000 AND 100000;

-- ORDER BY: sort results alphabetically
SELECT DISTINCT name
FROM   instructor
ORDER BY name ASC;  -- or DESC for reverse

FROM Clause, Joins & Rename

Combining data from multiple tables

🌎 Real Life Analogy

The FROM clause with a JOIN is like matching two spreadsheets together. Table A has employee details. Table B has which courses they teach. You want to combine them: "Show me teacher names AND the courses they teach." That requires linking both sheets by a common column (ID).

-- Join instructor and teaches tables on matching ID
SELECT name, course_id
FROM   instructor, teaches
WHERE  instructor.ID = teaches.ID;

-- Only Art department instructors
SELECT name, course_id
FROM   instructor, teaches
WHERE  instructor.ID = teaches.ID
  AND  instructor.dept_name = 'Art';

Rename with AS (Aliases)

🌎 Real Life Analogy

AS is like giving a nickname. In a meeting, instead of saying "the database administrator of the southern regional office," you say "Dave." Same concept — instructor AS T means "call this table T for short."

-- Find instructors earning more than any Biology instructor
SELECT DISTINCT T.name
FROM   instructor AS T, instructor AS S
WHERE  T.salary > S.salary
  AND  S.dept_name = 'Comp. Sci.';
🧠 Self Join

Notice we use the same table twice with different aliases. This is called a self-join — comparing rows within the same table. Useful for hierarchies like employee → supervisor relationships.

🔤

String Operations

Pattern matching and text manipulation

🌎 Real Life Analogy

String operations are like a smart search box. In Google, typing "intro*" might match "Introduction," "Introductory," "Intro to Physics." SQL's LIKE operator does the same inside your database.

Wildcards in LIKE

%

Matches any sequence of characters (including empty)

'%dar%' → matches "cedar", "Goldardson", "darko"

_

Matches exactly one character

'_ _ _' → matches exactly 3-character strings like "SQL", "ABC"

-- Find names containing "dar"
SELECT name FROM instructor WHERE name LIKE '%dar%';

-- Names starting with "Intro"
WHERE title LIKE 'Intro%'

-- Exactly 3 characters
WHERE code LIKE '_ _ _'

-- Escape: search for literal 100% (use backslash)
WHERE discount LIKE '100\%' ESCAPE '\'

-- String functions
UPPER('hello')         -- → 'HELLO'
LOWER('HELLO')         -- → 'hello'
LENGTH('SQL')          -- → 3
SUBSTRING('Database', 1, 4)  -- → 'Data'
CONCAT('Hello', ' World')    -- → 'Hello World'
🔵

Set Operations

UNION, INTERSECT, EXCEPT — combining query results

🌎 Real Life Analogy

Set operations are like Venn diagrams! Imagine two lists: List A = students who passed Math. List B = students who passed Science. UNION = everyone who passed either. INTERSECT = students who passed both. EXCEPT = students who passed Math but NOT Science.

🔵🟡

UNION

Combines results from both queries. Removes duplicates automatically.

🟢

INTERSECT

Only rows that appear in BOTH queries (the overlap).

🔴

EXCEPT

Rows in the first query that are NOT in the second.

-- UNION: courses in Fall 2017 OR Spring 2018
(SELECT course_id FROM section WHERE sem = 'Fall'   AND year = 2017)
UNION
(SELECT course_id FROM section WHERE sem = 'Spring' AND year = 2018);

-- INTERSECT: courses offered in BOTH semesters
(SELECT course_id FROM section WHERE sem = 'Fall'   AND year = 2017)
INTERSECT
(SELECT course_id FROM section WHERE sem = 'Spring' AND year = 2018);

-- EXCEPT: Fall 2017 but NOT Spring 2018
(SELECT course_id FROM section WHERE sem = 'Fall'   AND year = 2017)
EXCEPT
(SELECT course_id FROM section WHERE sem = 'Spring' AND year = 2018);

-- To KEEP duplicates, add ALL:
UNION ALL / INTERSECT ALL / EXCEPT ALL

Null Values

The mystery of "unknown" data

🌎 Real Life Analogy

NULL is like a blank field on a form. If a job application asks "Middle name" and someone leaves it blank — that's NULL. It doesn't mean their middle name is "" (empty) — it means we simply don't know or it doesn't exist. NULL ≠ 0, NULL ≠ "".

Key Rules

  • Any arithmetic with NULL = NULL
    5 + NULL → NULL
  • Any comparison with NULL = UNKNOWN
    NULL = NULL → UNKNOWN
  • Use IS NULL / IS NOT NULL to check

Boolean Logic with NULL

ExpressionResult
TRUE AND UNKNOWNUNKNOWN
FALSE AND UNKNOWNFALSE
TRUE OR UNKNOWNTRUE
FALSE OR UNKNOWNUNKNOWN
-- Find instructors with no salary entered
SELECT name
FROM   instructor
WHERE  salary IS NULL;

-- Find instructors who DO have a salary
SELECT name
FROM   instructor
WHERE  salary IS NOT NULL;

Aggregate Functions

Computing summary statistics on your data

🌎 Real Life Analogy

Aggregate functions are like the summary row at the bottom of a spreadsheet. Excel's SUM(), AVERAGE(), MAX(), MIN(), COUNT() — SQL has exact equivalents. Your HR team might ask: "What's the average salary per department? Who earns the most? How many employees do we have?"

➕

SUM()

Total of all values in a column

📈

AVG()

Average (mean) of all values

⬆️

MAX()

Largest value in the column

⬇️

MIN()

Smallest value in the column

🔢

COUNT()

Number of rows (or non-null values)

-- Average salary in Computer Science
SELECT AVG(salary)
FROM   instructor
WHERE  dept_name = 'Comp. Sci.';

-- Count distinct teachers in Spring 2018
SELECT COUNT(DISTINCT ID)
FROM   teaches
WHERE  semester = 'Spring' AND year = 2018;

-- GROUP BY: average salary per department
SELECT dept_name, AVG(salary) AS avg_salary
FROM   instructor
GROUP BY dept_name;

-- HAVING: only departments with avg > 42000
SELECT dept_name, AVG(salary) AS avg_salary
FROM   instructor
GROUP BY dept_name
HAVING AVG(salary) > 42000;
💡 WHERE vs HAVING

WHERE filters individual rows BEFORE grouping. HAVING filters groups AFTER aggregation. Think: WHERE picks which students take the test; HAVING picks which class averages are above passing grade.

🪆

Nested Subqueries

Queries inside queries — like Russian dolls

🌎 Real Life Analogy

A subquery is like answering a question to answer another question. "Which students scored above the class average?" — you first need to know what the average is (inner query), then use that result to find students above it (outer query). One question feeds into another.

IN / NOT IN

-- Courses taught in Fall 2017 AND Spring 2018
SELECT DISTINCT course_id
FROM   section
WHERE  semester = 'Fall' AND year = 2017
  AND  course_id IN (
    SELECT course_id
    FROM   section
    WHERE  semester = 'Spring' AND year = 2018
  );

-- Instructors who are not Mozart or Einstein
SELECT DISTINCT name
FROM   instructor
WHERE  name NOT IN ('Mozart', 'Einstein');

SOME / ALL — Comparing to a Set

🌎 Real Life Analogy

SOME = "better than at least one person in the group." If you scored higher than any student in Class B, you used SOME. ALL = "better than everyone in the group." You must beat every single student in Class B.

-- Salary > at least one Biology instructor (SOME = > MIN)
SELECT name
FROM   instructor
WHERE  salary > SOME (
    SELECT salary FROM instructor WHERE dept_name = 'Biology'
);

-- Salary > ALL Biology instructors (ALL = > MAX)
SELECT name
FROM   instructor
WHERE  salary > ALL (
    SELECT salary FROM instructor WHERE dept_name = 'Biology'
);
ClauseEquivalent ToMeaning
> SOME> MIN()Greater than the smallest value
< SOME< MAX()Less than the largest value
= SOMEINMatches at least one value
> ALL> MAX()Greater than every value
< ALL< MIN()Less than every value

EXISTS & NOT EXISTS

-- Courses in Fall 2017 that also ran in Spring 2018
SELECT course_id
FROM   section AS S
WHERE  semester = 'Fall' AND year = 2017
  AND EXISTS (
    SELECT *
    FROM   section AS T
    WHERE  semester = 'Spring' AND year = 2018
      AND  S.course_id = T.course_id
  );

WITH Clause — Temporary Named Results

🌎 Real Life Analogy

WITH is like creating a temporary sticky note with a calculated result, then using that result in your main question. "First, figure out the max budget across all departments. Then, show me which departments have that budget."

-- WITH clause: find departments with the maximum budget
WITH max_budget(value) AS (
    SELECT MAX(budget) FROM department
)
SELECT department.name
FROM   department, max_budget
WHERE  department.budget = max_budget.value;

INSERT, DELETE, UPDATE

Adding, removing, and changing data

🌎 Real Life Analogy

These three operations are like managing a physical address book: INSERT = writing a new contact. DELETE = erasing a contact. UPDATE = crossing out an old phone number and writing the new one. All essential!

INSERT

-- Add a new course
INSERT INTO course
VALUES ('CS-437', 'Database Systems', 'Comp. Sci.', 4);

-- Explicit columns (best practice)
INSERT INTO course (course_id, title, dept_name, credits)
VALUES ('CS-437', 'Database Systems', 'Comp. Sci.', 4);

-- Insert NULL for unknown value
INSERT INTO student
VALUES ('3003', 'Green', 'Finance', NULL);

DELETE

-- Delete all instructors
DELETE FROM instructor;

-- Delete only Finance department instructors
DELETE FROM instructor
WHERE  dept_name = 'Finance';

-- Delete instructors earning below average (computed once first)
DELETE FROM instructor
WHERE  salary < (SELECT AVG(salary) FROM instructor);

UPDATE

-- Give everyone a 5% raise
UPDATE instructor
SET    salary = salary * 1.05;

-- Only give raise to those earning < 70000
UPDATE instructor
SET    salary = salary * 1.05
WHERE  salary < 70000;

-- CASE: tiered raises (3% for high earners, 5% for others)
UPDATE instructor
SET    salary = CASE
    WHEN salary <= 100000 THEN salary * 1.05
    ELSE                         salary * 1.03
END;
⚠️ Order Matters with UPDATE

If you run two UPDATE statements separately — "give 3% to those above 100k" then "give 5% to those below 100k" — someone who was just raised from 99k to 102k now gets BOTH raises! The CASE statement handles this safely in a single pass.
