3. Which one of followings describe "Atomicity of updates" correctly? ( )
A. Failures may leave database in an inconsistent state with partial updates carried out
B. Concurrent access needed for performance
C. Hard to provide user access to some, but not all, data
D. Integrity constraints become "buried" in program code rather than being stated explicitly
✓ ANSWER: A
Explanation: Atomicity ensures that either all operations in a transaction complete or none do. Without atomicity, failures could leave the database in an inconsistent state with partial updates. This is the core definition of the atomicity problem.

Explanation: An Instance is the actual data (content) stored in the database at any given time. Schema defines the structure, while instance represents the actual data values.

18. Using the university schema, write the following queries in SQL.
Find the ID and name of each student who has not taken any course offered before 2017. (Use Select, From, Where)
STEP 1: Understanding the Problem
  • We need: Student ID and Name
  • Condition: Students who have NOT taken courses from before 2017
  • Approach: Use NOT IN with a subquery
STEP 2: SQL Solution
SELECT DISTINCT s.ID, s.name FROM student s WHERE s.ID NOT IN ( SELECT t.ID FROM takes t JOIN course c ON t.course_id = c.course_id WHERE c.year < 2017 );
19. Using the university schema, write an SQL query to find the names of those departments whose budget is higher than that of Philosophy. List them in alphabetic order.
STEP 1: Understanding the Problem
  • We need: Department names
  • Condition: Budget > Philosophy's budget
  • Sort: Alphabetically (A to Z)
  • Approach: Subquery to get Philosophy's budget first
STEP 2: SQL Solution
SELECT d.dept_name FROM department d WHERE d.budget > ( SELECT d2.budget FROM department d2 WHERE d2.dept_name = 'Philosophy' ) ORDER BY d.dept_name ASC;
20. Using the university schema, write an SQL query to find the IDs of those students who have retaken at least three distinct courses at least once (i.e, the student has taken the course at least two times).
STEP 1: Understanding the Problem
  • Retaken = taken the course 2 or more times
  • We need: Student IDs
  • Condition: Taken at least 3 courses that were retaken
  • Approach: GROUP BY student, use HAVING with COUNT conditions
STEP 2: SQL Solution
SELECT t.ID FROM takes t GROUP BY t.ID HAVING COUNT(DISTINCT t.course_id) >= 3 AND ( SELECT COUNT(DISTINCT course_id) FROM takes t2 WHERE t2.ID = t.ID GROUP BY course_id HAVING COUNT(*) >= 2 ) >= 3;
21. Using the university schema, write an SQL query to find the ID and title of each course in Comp. Sci. that has had at least one section with afternoon hours (i.e., ends at or after 12:00). (You should eliminate duplicates if any.)
STEP 1: Understanding the Problem
  • We need: Course ID and Title
  • Condition: Department = 'Comp. Sci.' AND time_end >= 12:00
  • Afternoon: 12:00 (noon) or later
  • Eliminate duplicates: Use DISTINCT
STEP 2: SQL Solution
SELECT DISTINCT c.course_id, c.title FROM course c, section s WHERE c.course_id = s.course_id AND c.dept_name = 'Comp. Sci.' AND s.time_end >= '12:00';
22. Using the university schema, write an SQL query to find the ID of each student who has never taken a course at the university. Do this using no subqueries and no set operations (use an outer join).
STEP 1: Understanding the Problem
  • We need: Student IDs
  • Condition: Students with NO courses taken
  • No subqueries allowed!
  • Use: LEFT OUTER JOIN + IS NULL check
STEP 2: SQL Solution
SELECT s.ID FROM student s LEFT OUTER JOIN takes t ON s.ID = t.ID WHERE t.ID IS NULL;
23. Explain why, when a manager, say Satoshi, grants an authorization, the grant should be done by the manager role, rather than by the user Satoshi.

1. Separation of Duties:

Authorization is tied to the responsibility (manager role), not the individual person.

2. Accountability & Continuity:

If Satoshi leaves, the manager role remains with all its permissions. The new manager inherits them automatically.

3. Consistency:

All users with the manager role have the same permissions, preventing inconsistencies across the organization.

4. Security Management:

Permissions are managed at the role level, making auditing and changes easier. When Satoshi leaves, revoke the role rather than individual permissions.

5. Scalability:

Permission changes apply to all role members automatically without individual modifications.
